## 知見メモ

### 1章

- **トークンの仕組み**：GPT-4の平均トークン長は単語の3/4。モデルが扱うすべてのトークンのセットを「語彙（vocabulary）」という。トークン単位で管理する理由は、"cooking" を "cook" + "ing" のように分割して管理するため
- **機械学習の理論理解は依然重要**：利用できるツールの適用範囲を広げ、モデルが期待通りに機能しない際のトラブルシューティングにも役立つ
- **80/100%の法則**：顧客に対してAI開発のコスト感を正しく伝えるための説明材料として活用する
- **ビジネス↔ML指標マッピング**：AIの進化に関わらず変わらない本質。顧客への提案・説明で使える

### 2章

- **サンプリング戦略**（temperature / top-k / top-p）：AIシステム設計に直結する最も過小評価されている概念。詳細は章別詳細メモ参照
- **MoE（Mixture of Experts）**：推論時は一部のエキスパートのみ起動するため速度・コストが変わる。代表例はMixtral 8x7B
- **事後学習の2ステップ**：①教師ありファインチューニング → ②選好ファインチューニング
- **ハイパーパラメータ転移**：大規模モデルではHPOを複数試せないため、小さいモデルで検証したハイパラを大規模モデルに転用する手法（スケーリングの外挿）
- **テスト時計算**：AIに複数の回答を生成させ、別のAIに最良案を選ばせるだけで単一出力より精度が上がる
- **アライメント過学習の副作用**：アライメント学習をやりすぎると、政治・宗教的見解やシャットダウン回避の意志を表現するようになるという論文がある
- **ハルシネーションの2大仮説**：詳細は章別詳細メモ参照

### 評価章

- **評価の3分類**：①機能正確性（テストコードと同概念）、②類似スコア（連続値で評価）、③AI as a Judge（AIに評価させる）
- **AI as a Judge**：重要。評価基準・タスク・スコアリング（離散値が得意）をプロンプトに記述。**必ず異なるモデルを使うこと**（自己バイアスがあるため）
- **AI as a Judge のバイアス3種**：自己バイアス・先頭位置バイアス・冗長性バイアス。詳細は章別詳細メモ参照
- **ユーザー評価（good/bad）の限界**：ユーザーがすでに知っている業務にしか有効でない
- **比較 vs 絶対パフォーマンス**：絶対パフォーマンスが実務上は最重要。ただし比較の方が測定しやすい
- **複数モデルの効率的な評価**：ランキング＋マッチングアルゴリズムの適切な選択が必要

### 4章

- **評価駆動開発**：運用後も評価を継続し、サービスが具体的な価値向上に貢献しているか常に確認する開発スタイル
- **AIアプリ固有の評価基準リストを作ることから始める**（超重要）
- **評価基準の4分類**：①ドメイン固有能力、②生成能力、③指示追従能力、④コストとレイテンシー（詳細は章別詳細メモ参照）
- **モデル選択はループで回す**：内製/外注の判断 → 公開ベンチマーク → タスク固有の評価 → 本番モニタリング
- **オープンソース vs API**：基本はAPIでよい。外部に出せないデータ・未対応の制約・レイテンシー問題が出た場合にオープンモデルを検討。ただし制約条件自体を変えることも先に検討すべき
- **APIは黙って変更される**：評価駆動開発で自動テストを即時実行できる体制が必須
- **評価パイプライン構築の4ステップ**：詳細は章別詳細メモ参照
- **評価データのアノテーション**：属性ラベルを付けてサブセット分析することで、バイアスや改善点、シンプソンのパラドックスなどを検出できる
- **評価データの量**：最低300サンプル、合成データの場合は1000サンプル以上

### 5章

- **プロンプトテンプレートはモデル・バージョンごとに異なる**：最初に必ず確認。意図しない精度劣化を招き、後から発見が困難
- **モデルはプロンプトの最初と最後が得意**：中央の情報は見落とされやすい
- **CoT・自己批判**：「段階的に考えて」「その回答は正しいか？」を加えるだけで精度改善できる
- **プロンプトは専用ツールでバージョン管理必須**：LangSmith, PromptLayer, Helicone, Humanloop など
- **プロンプト攻撃対策**：侵害率（防げなかった割合）・誤拒否率で評価。既知攻撃の自動テストツールも活用（Azure/PyRIT, garak 等）
- **RAG**：リトリーバー（検索器）＋生成モデルの構成。検索はタームベース（BM25等）と埋め込みベース（セマンティック）がある。詳細は章別詳細メモ参照
- **ANNライブラリ**：FAISS, ScaNN, Hnswlib。評価軸は再現率・QPS・ビルド時間・インデックスサイズ
- **エージェントのメモリ3種**：①内部知識（学習データ）、②短期記憶（コンテキスト内）、③長期記憶（RAG等）
- **ReACTパターン**：PLAN → Action → Reflection のループがエージェント設計の基本
- **プランニングの評価**：（タスク、ツールインベントリ）のペアでデータセットを作り定量評価できる

### 7章

- **ファインチューニングは基本しないほうがよい**：広範タスクに使うモデル・メンテできない場合は避ける。ベースモデルも日々進化するため継続的更新が必要
- **ファインチューニングするケース**：性能の根本改善・バイアス低減（主に小さいモデル）・自社ホスティング
- **PEFT が主流**：フルファインチューニングはメモリ消費が大きいため、パラメータ効率の良い手法が発展。LoRA が人気
- **モデルマージ**：各デバイスでローカル学習したモデルをマージすることで、データを外部に出さずに連合学習を実現できる

## 章別詳細メモ

### 2章：モデルアーキテクチャと学習

#### サンプリング戦略（temperature / top-k / top-p）

生成AIが出力トークンをどう選ぶかという戦略で、AIシステム構築において重要。

- **temperature**：softmax前にスコアをT（0〜∞）で割ることで確率の差を操作する
  - T が高い → 確率差が縮まる → 多様な表現が出やすい（創造的）
  - T が低い → 確率差が広がる → 確率最大のトークンに集中（単一的・確定的）
- **top-k**：確率上位k個のトークンの中から確率に従ってサンプリングする
- **top-p**：累積確率がpを超える最小の集合の中から確率的に選ぶ（候補数が動的に変わる）

#### ハルシネーションの原因仮説と対策

**仮説1：雪だるま式ハルシネーション**
推論を重ねるたびに、前のステップで誤った仮定を事実と見なしてしまい、妄想が雪だるま式に膨らむ。
→ 対策：強化学習でユーザーが提供したプロンプトとモデルが生成したトークンを区別するよう学習させる

**仮説2：知識ラベル不一致**
モデルの内部知識とラベラーの知識が一致しないため、ラベラーが誤った情報（ハルシネーション）を正として学習させてしまう。前提として「LLMは自分が何を知っているかを自覚している」という仮説に基づく。
→ 対策①：各応答についてモデルに根拠となる情報源を取得させる
→ 対策②：強化学習で作り話に対して厳しい報酬関数を設計する

### 評価章：AI評価の方法論

#### 参照応答を用いた評価と一致率の測定方法

参照応答（あらかじめ用意した正解文）との一致率を測る方法は4種類ある。参照応答はAIに作らせて人間がチェックするケースが多く、複数になることが一般的。

| # | 方法 | 概要 |
|---|------|------|
| 1 | 人間による判断 | 2つのテキストが一致するかを評価者に判断させる |
| 2 | 完全一致 | 参照応答のいずれかと完全に一致するか |
| 3 | 字句的類似性 | 単語レベルでどの程度重なっているか |
| 4 | 意味的類似性 | 意味的にどの程度近いか（埋め込みベクトルで計測） |

**字句的類似性の主な指標**：BLEU, ROUGE, METEOR++, TER, CIDEr など。各指標は重複部分の計算方法が異なる。参照応答が網羅的でないと実態より悪いスコアが出やすい点に注意。

**意味的類似性の主な指標**：BERTScore, MoverScore など。代表的な埋め込みモデルは Google BERT, OpenAI CLIP, OpenAI Embeddings API, Cohere Embed v3。

#### AI as a Judge のバイアス

| バイアス | 内容 |
|----------|------|
| 自己バイアス | 自身（同じモデル）の出力を高く評価しやすい |
| 先頭位置バイアス | 選択肢・リストの先頭を高く評価しやすい（人間はリーセンシーバイアスで末尾を評価しやすい） |
| 冗長性バイアス | 事実誤認でも長い応答を好む。2倍以上の差があればほぼ常に長いほうを高評価。モデル性能向上で改善傾向あり |

→ AI as a Judge 単独での評価は限界があり、厳密な評価方法や人間評価と組み合わせるべき。

### 4章：AIシステムの評価

#### 評価基準の4分類

**① ドメイン固有能力**
AIが実行する業務特有の思考能力そのもの。一般的には完全一致評価で測定する。よくある手法は多肢選択問題（MCQ）でクローズドエンドに持ち込む方法。ただし、内容に影響がない範囲でプロンプトを少し変えるだけで回答が変わる現象が観測されており、プロンプト感度の制御が重要。クローズドエンドな評価が本当に適切かはまだ結論が出ていない。

**② 生成能力**
生成された文章そのものの品質。コーディングができても人間が読めなければ意味がない、といった観点。評価項目は実務ごとに設計が必要。主な観点：

- **事実整合性**：ローカル（社内文書・プロンプト内容への整合）とグローバルの2種類。RAGシステムで特に重要
  - 検証方法1：自己検証（SelfCheckGPT など）→ コストが高い
  - 検証方法2：知識拡張検証（SAFE）→ 応答を複数の事実に分割し、各事実にファクトチェックを実施して裏付け件数を測定
- **安全性**：ポリシーへの準拠

**③ 指示追従能力**
与えられた指示にどれだけ従っているかを測定する。

**④ コストとレイテンシー**（超重要）

| メトリック | 内容 |
|-----------|------|
| コスト | 出力トークン当たりのコスト |
| スケール | 1分あたりのトークン数 |
| レイテンシー | 最初のトークンまでの時間 / クエリ当たりの合計時間 |
| モデル品質 | Eloスコアなど |

#### 評価パイプライン構築の4ステップ

**ステップ1：コンポーネントの分解**
システム内のすべてのコンポーネントをターン毎・タスク毎に分解して評価する。タスク分解できるならそうする。

**ステップ2：評価ガイドラインの作成**（最重要）
何をもって良い・悪いとするかを定義する。定義が決まったら評価指標を作り、ビジネス指標と結びつける。

**ステップ3：評価指標とデータの定義**
- 決定論的タスク（分類など）：logprobs で回答感度を測定
- それ以外：AI as a Judge
- 評価データにアノテーション（属性ラベル）を付け、サブセット分析できるようにする
  - 検出できるもの：マイノリティへのバイアス、属性固有の難しさ、改善点、シンプソンのパラドックス
  - 属性例：ユーザーグループ別、使用デバイス別、間違いやすさ別、アプリ対象外かどうか

**評価データの品質・量の確認**
- 品質：ブートストラップで評価データを複製し、結果がブレるなら信頼できない
- 量：最低300サンプル（合成データの場合は1000サンプル以上）

**ステップ4：評価パイプライン自体の検証**

| 検証観点 | 内容 |
|---------|------|
| 妥当性 | よりよい回答が実際に高スコアになっているか |
| 信頼性 | ブートストラップで平均・分散を確認。AI as a Judge の temperature は必ず0に |
| 指標間の相関 | 完全相関なら片方は不要。無相関なら洞察があるか片方が信頼できないかのどちらか |
| コスト・レイテンシーへの影響 | 評価パイプライン自体がどの程度負荷になるか |

評価パイプラインは運用しながら改善する。ただし一貫性は保つこと。使用する変数はすべてログに残す。

### 7章：ファインチューニング

ファインチューニングはメモリ消費が大きいため様々な軽量化手法が開発されている。

**部分ファインチューニングの2アプローチ**

| アプローチ | 概要 | 代表手法 |
|-----------|------|---------|
| アダプターベース（PEFT） | アーキテクチャにアダプター層を追加し、そこだけ学習 | LoRA |
| ソフトプロンプトベース | ハードプロンプト（人間の言葉）を学習可能なベクトルに変換し、アーキテクチャの途中に差し込む | Prefix Tuning 等 |

**モデルマージ（連合学習）**：モデルXを各ユーザーデバイスにコピー → デバイス上のローカルデータで学習 → 各デバイスのモデルをマージ。データをデバイス外に出さずにモデルを改善できる。

### 5章：プロンプトエンジニアリング・RAG・エージェント

#### プロンプトの基本構成

| 用語 | 説明 |
|------|------|
| プロンプト | モデルへの入力全体 |
| コンテキスト | タスク実行のために提供される情報（few-shot例示など） |
| システムプロンプト | タスクの説明・指示 |
| ユーザープロンプト | 実際のタスク |

プロンプトテンプレート（システムプロンプトとユーザープロンプトを1つに結合する形式）はモデル・バージョンごとに異なる。意図しない精度劣化を招き、応答自体はそれっぽいため後から発見が非常に困難。**最初に必ず確認する。**

#### プロンプト攻撃の種類とリスク

攻撃種類：プロンプト抽出、脱獄（jailbreak）、プロンプトインジェクション、情報抽出

リスク：リモートコード実行、データ漏洩、社会的危害、偽情報、サービス妨害、ブランドリスク

対策ツール（既知攻撃の自動実行）：Azure/PyRIT, 1eondz/garak, greshake/llm-security, CHATS-lab/persuasive_jailbreaker

評価指標：**侵害率**（防げなかった割合）・**誤拒否率**（誤って拒否した割合）= precision/recall と同概念

#### RAGの検索アルゴリズム

**分類1：タームベース vs 埋め込みベース（セマンティック）**
- タームベース：TF-IDF, BM25（TF-IDFをドキュメント長で正規化したもの）。単語の分割にはn-gramを使用（"hot dog" のような複合語に対応）
- 埋め込みベース：クエリと文書を埋め込みベクトルに変換して類似度を計算
- スパース埋め込み（SPLADE等）も存在

**分類2：スパース検索 vs デンス検索**

**近似最近傍探索（ANN）**：k-NNは低速なため実務ではANNを使用。主要ライブラリ：FAISS, ScaNN, Hnswlib

ANNの評価軸：

| 軸 | 内容 |
|----|------|
| 再現率 | 見つけられた最近傍の割合 |
| QPS | 1秒あたりのクエリ処理数 |
| ビルド時間 | インデックス構築に要する時間 |
| インデックスサイズ | メモリ・ストレージ使用量 |

**RAG評価指標**：コンテキスト精度（precision）・コンテキスト再現率（recall）。再現率の計算には全評価データへのアノテーションが必要で工数が大きい。

**マルチモーダルRAG**：
- 画像取得：画像のキャプションを埋め込み → クエリに近いキャプションを検索して対応画像を返す
- クエリに画像も含む場合：CLIP等のマルチモーダル埋め込みでキャプション＋画像を埋め込んで検索

#### エージェントのリフレクション・プランニング

**ReACTパターン（Yaoら提案）**：推論（PLAN）と行動を交互に行う手法がエージェントの基本パターンに。PLAN → Action → Reflection をループさせる構造。

**リフレクション**：現在うまくいっているか・いないかをモデルに考えさせるコンポーネント。

**プランニングの評価方法**：

（タスク、ツールインベントリ）のペアで構成されるデータセットを作成し、エージェントにK個のプランを生成させて以下を計測：

- 有効なプランの数
- 有効なプランを得るために必要な平均生成数
- ツール呼び出しのうち有効なものの割合
- 無効なツールの呼び出し頻度
- 有効なツールが無効なパラメータ・不正な値で呼ばれる頻度
