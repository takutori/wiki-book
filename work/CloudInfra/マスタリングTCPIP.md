## 主張

インターネット通信は「層ごとに役割を分担する」という設計原則（OSI参照モデル／TCP/IP）によって成り立っており、各層のプロトコルがどう連携するかを理解することが、ネットワーク全体を把握する鍵になる。

### 根拠・論理構造

- 通信機能を7層（OSI）または4層（TCP/IP）に分割することで、各層が独立して設計・変更でき、異なる機器・OS間でも相互接続が可能になる
- 物理層〜データリンク層（Ethernet, MACアドレス）が同一ネットワーク内の転送を担い、ネットワーク層（IP）がネットワーク間の転送を担うという明確な役割分担がある
- IPはコネクションレス・ベストエフォートであり、信頼性はトランスポート層（TCP）が補完するという分業構造になっている
- DNS・ARP・DHCP・NATなど周辺プロトコルが連携することで、IPアドレスの自動設定・名前解決・アドレス変換が実現し、現代のインターネットが動いている
- IPv4のアドレス枯渇という具体的な問題がIPv6誕生の背景にあり、技術は現実の課題に応じて進化している

## 知見メモ

### 第1章：ネットワーク基礎知識

**OSI参照モデル**

各社がプロトコルを独自に作っていたため異なる機器間通信が困難だった→ISOが通信を標準化するためOSI参照モデルを提唱。通信機能を7層に分割し、上下層のやり取りはインターフェース、同層間のやり取りはプロトコルで行う。

| 層 | 名前 | 役割 | 例 |
|--:|---|---|---|
| 7 | アプリケーション層 | アプリが使う通信ルール | HTTP, DNS, SMTP |
| 6 | プレゼンテーション層 | データ形式変換・暗号化・圧縮 | TLS, 文字コード, JSON変換 |
| 5 | セッション層 | 通信セッションの開始・維持・終了 | セッション管理 |
| 4 | トランスポート層 | アプリ間通信・ポート・信頼性制御 | TCP, UDP |
| 3 | ネットワーク層 | IPアドレスでネットワーク間を届ける | IP, ルーター |
| 2 | データリンク層 | 同じLAN内で機器に届ける | Ethernet, WiFi, MAC, スイッチ |
| 1 | 物理層 | 電気・光・電波として送る | LANケーブル, 光, 無線電波 |

送信側：アプリケーション層→物理層（上から下へ）
受信側：物理層→アプリケーション層（下から上へ）

**メッセージ送信の例（AさんからBさんへ）**

| 層 | 処理内容 |
|---|---|
| アプリケーション層 | 宛先・メッセージ内容などデータを整理 |
| プレゼンテーション層 | 機器・アプリ間で異なるデータ表現を統一フォーマットに変換（例：Shift_JIS→UTF-8） |
| セッション層 | コネクションの確立・切断の計画 |
| トランスポート層 | 通信路の確保・コネクション確立の実行 |
| ネットワーク層 | IPアドレスを特定し、ネットワーク間でデータを配達 |
| データリンク層・物理層 | MACアドレス宛に信号として伝送 |

**ネットワーク接続ハードウェア**

| 機器 | 動作する層 | 役割 |
|---|---|---|
| ネットワークインターフェース（NIC） | L1 | PCをネットワークに接続する口。最近のPCは無線LAN内蔵が多い |
| リピーター | L1（物理層） | 電気・光信号を受信して増幅・波形整形し延長する |
| ブリッジ / L2スイッチ | L2（データリンク層） | MACアドレスでネットワーク同士を接続。データを一時蓄積後に送出。FCSで誤り検出 |
| ルーター / L3スイッチ | L3（ネットワーク層） | IPアドレスでネットワーク間を接続・中継 |
| L4-7スイッチ | L4〜L7 | ポート番号・アプリ情報に基づいて配送。ロードバランサーが代表例 |
| ゲートウェイ | L4〜L7 | 異なるプロトコル間でデータを変換して中継。プロキシサーバーもここに含まれる |

### 第2章：TCP/IP 基礎知識

**インターネットの構造**
- 組織ネットワーク → 地域ネットワーク → インターネットと階層的に接続
- **バックボーン**：基幹ネットワーク　／　**スタブ**：末端ネットワーク
- ネットワーク同士は**NOC**で接続し、**IX**で対等に相互接続する

**TCP/IP 各層の詳細**

| 層 | 役割・特徴 |
|---|---|
| 物理層 | データを転送するハードウェア |
| ネットワークインターフェース層 | OSとハードウェアの橋渡しをするソフトウェア |
| インターネット層（IP） | IPアドレスでパケットを転送。異常があっても放置（ベストエフォート）。ICMPが異常を送信元に通知。ARPでIPアドレス→MACアドレスを取得 |
| トランスポート層 | アプリ間通信をポート番号で識別。TCP/UDPの2プロトコル |

**TCPとUDP**

| | TCP | UDP |
|---|---|---|
| 接続方式 | コネクション型 | コネクションレス型 |
| 信頼性 | あり（パケットロス・順序入れ替えを自動解決、帯域制御・輻輳制御あり） | なし（到達確認なし） |
| 不向きなケース | 処理数が多く転送量が少ない場合、一定間隔で決まった量を送るビデオ通話など | — |

**アプリケーション層の主なプロトコル**

| プロトコル | 用途 |
|---|---|
| HTTP | WebブラウザとサーバーのWWW通信 |
| SMTP | 電子メール送信 |
| FTP | ファイル転送 |
| TELNET / SSH | 遠隔ログイン |
| SNMP | ネットワーク管理。エージェント（ルータ等）とマネージャー間の通信に使用 |

**「こんにちは」を送る通信フロー**
1. アプリがUTF-8等で符号化
2. TCPがコネクションを確立
3. IPが宛先IPアドレスを含むパケットを生成
4. ARPでMACアドレスを取得
5. NI層が実際に信号として送出

### 第3章：データリンク

- **データリンク**：具体的な通信手段（イーサネット、無線LAN等）を指す一般的な用語
- コンピュータ上の情報は0/1だが、実際の通信媒体では電圧変化・光の点滅・電波の強弱で伝わる。この変換は物理層が担当
- **MACアドレス**：データリンクに接続したノードの識別に使用。ただし階層構造がないため特定しにくい→階層構造を持つIPアドレス（ネットワーク層）が重要になる

**通信方式**
- **半二重通信**：送受信を同時にできない（例：無線トランシーバー）
- **全二重通信**：送受信を同時に行える（例：電話）
- **CSMA/CD**：イーサネットが採用する方式。通信可能かを確認してから媒体を独占してデータを送る。ケーブル内部に複数の線があり、送信・受信を同時に行える

**関連技術**
- **LLDP**：ネットワーク接続機器の情報を収集する仕組み。定期的にマルチキャストMACアドレスへホスト名・機器情報・ポート情報を送信する
- **VLAN**：ネットワークを仮想的に分割。タグVLANでデータの所属VLANを識別し、物理的なケーブル差し替えなしにネットワーク所属を変更できる

**イーサネットの種類（例）**
- `10BASE-F`：光ファイバー
- `100BASE-TX`：ツイストペアケーブル

### 第4章：IP

**IPの役割**
- IPはインターネット層のプロトコル。IP・ICMPの2つで構成される
- 3つの役割：①IPアドレスの管理、②終点ホストまでのパケット配送、③パケットの分割・再構築
- コネクションレス型：相手の受信可否を確認せずに送信
- ネットワーク層が通信の全工程を管理し、データリンク層が各工程の通信を担当

**IPv4**
- 32ビット／4オクテット（例：192.168.1.1）。約43億台（2³²）まで接続可能
- IPアドレス = **ネットワーク部** + **ホスト部**。境界はサブネットマスクで表す（ネットワーク部=1、ホスト部=0）
- ホストアドレスのすべて0 → ネットワークアドレス（割り当て不可）
- ホストアドレスのすべて1 → ブロードキャストアドレス（割り当て不可）
- **プライベートIPアドレス**：一定範囲内で自由に設定可。NATによりグローバルIPアドレスと変換してインターネット接続も可能。現在はルータ・公開サーバーにのみグローバルIPアドレスを設定するのが一般的

**IPv6**
- IPv4のアドレス枯渇を解決するために登場。IPv4の問題点も同時に改善
- 128ビット長：約3.4×10³⁸台（天文学的な数）
- 主な改善点：アドレス空間の拡大・経路制御表の集約・パフォーマンス向上・プラグ＆プレイ必須化・認証/暗号化機能・マルチキャスト・Mobile IPのサポート

### 第5章：IPに関する技術

**DNS**
- ホスト名（ドメイン名）をIPアドレスに変換するプロトコル
- ドメインは階層構造：トップレベルドメイン（.com, .jp）→ 国コードTLD → 分野別TLD
- **ネームサーバー**：ドメイン名を管理するホスト/ソフトウェア。ルートネームサーバーを頂点に階層構造
- **リゾルバ**：DNSに問い合わせを行うホスト/ソフトウェア

**ARP / RARP / GARP**
- **ARP**：IPアドレス→MACアドレスの解決。ブロードキャストで「このIPを持つのは誰？」と問い合わせ、該当機器がMACアドレスを返す。結果はARPテーブルにキャッシュ
- **RARP**：MACアドレス→IPアドレスの逆引き。DHCP等で自分のIPがわからない場合に使用
- **GARP**：自分のIPアドレスに対するMACアドレスを問い合わせる。IPアドレスの重複チェックに利用

**ICMP**

| コード | メッセージ | 意味 |
|---|---|---|
| 0 | Network Unreachable | 宛先ネットワークへの経路情報がない |
| 1 | Host Unreachable | 宛先コンピュータがネットワークに接続されていない |
| 4 | Fragmentation Needed and DF set | 経路MTU探索で使用 |
| 11 | Time Exceeded | パケットが長時間（ループ等）流れ続けるのを防ぐ |

**DHCP**
- IPアドレスの自動設定・一括管理を行う
- IPアドレス付与前にICMPエコー要求とARP要求を送信し、重複がないことを確認してから払い出す

**NAT / NAPT**
- **NAT**：プライベートIPアドレス↔グローバルIPアドレスの変換
- **NAPT**：アドレス変換に加えてTCP/UDPのポート番号も付け替える。複数端末で1つのグローバルIPを共有できる

### 第6章：TCPとUDP（途中まで）

- トランスポート層のプロトコルはTCP・UDPの2種類。どちらを使うかはIPヘッダのプロトコル番号で定義
- TCP・UDPともにポート番号でデータを渡すアプリを指定する
- **使い分け**：信頼性重視→TCP、リアルタイム性重視→UDP
- **ポート番号**：ウェルノウンポート（HTTP, DNS, DHCPなど標準定義）とOSが自動割り当てするダイナミックポートがある
