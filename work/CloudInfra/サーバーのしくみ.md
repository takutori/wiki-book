## 主張

サーバーとは「アプリケーションを動かす基盤」であり、その役割・種類・通信の仕組み・障害対策・運用管理を体系的に理解することで、ITインフラ全体の構造を把握できる。

### 根拠・論理構造

- サーバーは受動的（要求に応答）・能動的（自ら処理を起動）・高性能活用型の3種に分類される
- クラウドの提供形態（SaaS/PaaS/IaaS）はサーバーの管理範囲の違いであり、利用者の責任境界を決める
- 通信はTCP/IP 4層モデルで成立し、各層が「何を・どこへ・どうやって」送るかを分担している
- 可用性はクラスタリング・ロードバランシング・RAID・チーミングなど複数の技術レイヤーで担保する
- 運用品質はSLA（可用性・MTTR）という指標で定量化される

## 知見メモ

### サーバーの種類

- **受動的サーバー**：クライアントからの要求に応答する（例：メールサーバー、プリントサーバー）
- **能動的サーバー**：処理のタイミング・内容をサーバー自身が決定する（例：RPAサーバー、BPMSサーバー）
- **高性能活用型**：AIサーバー、ビッグデータサーバーなど

### クラウドの提供形態

| 形態 | 管理範囲 | 例 |
|---|---|---|
| SaaS | すべてベンダー管理。ユーザーはアプリを使うだけ | Gmail、Slack、Notion |
| PaaS | DBやミドルウェア・開発環境を含む。アプリはユーザーが用意 | Vercel、Heroku、Azure App Service |
| IaaS | OSのみ提供。ミドルウェア・アプリはユーザーがインストール | AWS EC2、Azure VM、Google Compute Engine |

### 通信の仕組み

**IPアドレスとMACアドレス**
- IPアドレス：通信の最終的な宛先（住所）。ルーターが道案内
- MACアドレス：機器固有の番号。次ホップへの実際の転送に使用
- MACアドレスが必要な理由：WiFi・Ethernet・光回線・5Gなど通信方式が異なるため、その差異を吸収する層として機能する

**TCP/IP 4層モデル**

| 層 | 役割 | 主なプロトコル・技術 |
|---|---|---|
| アプリケーション層 | どの要件で通信するか | HTTP、HTTPS、DNS、SMTP |
| トランスポート層 | どのアプリに届けるか（ポート番号で管理） | TCP、UDP |
| インターネット層 | どのIPアドレスに届けるか | IP |
| ネットワークインターフェース層 | 実際にどう送るか | Ethernet、WiFi、MACアドレス |

### 主なサーバーの役割

- **SMTPサーバー**：メール送信・受信窓口
- **POP3サーバー**：メール受信
- **Webサーバー**：Webコンテンツ提供。DNS→IPアドレス変換を経てたどり着く
- **DNSサーバー**：ドメイン名→IPアドレス変換（URL・メールアドレス両方）
- **SSOサーバー**：一度のログインで複数アプリにアクセス可能にする
  - リバースプロキシ方式：ログインを代行
  - エージェント方式：各アプリと緻密に連携
- **プロキシサーバー**：通信を中継。閲覧制限・保護・記録・高速化・送信元隠蔽を担う
- **運用監視サーバー**：リソース監視・ヘルスチェック

**SSL/TLS（通信暗号化）**
サーバーが証明書を送付 → ブラウザが検証 → 共通鍵を生成 → 以降は共通鍵で暗号化通信

### 自動化・業務処理

- **RPA**：画面操作を含む人の作業を自動実行。画面レイアウト変更に弱い（例：Power Automate）
- **BPMS**：RPAの処理を複数つなげてワークフロー化し、フロー・業務状態を一元管理
- **IoT**：データ取得はデバイス主体でサーバーへ送信する方式が主流

### 可用性・障害対策

**サーバー冗長化**

| 項目 | クラスタリング | ロードバランシング |
|---|---|---|
| 主目的 | 高可用性・冗長化・フェイルオーバー | 負荷分散・障害サーバーの切り離し |
| 障害時の動き | 別ノードでサービスを再起動・引き継ぎ | 異常サーバーに通信を送らない |
| 代表例 | Windows Failover Cluster、Kubernetes | AWS ALB/NLB、Nginx、HAProxy |

**ネットワーク冗長化**
- **チーミング**：NIC複数化による冗長化・負荷分散。フォルトトレランス（クラスタリング相当）とロードバランシングの2方式がある

**ストレージ冗長化（RAID）**

| レベル | 方式 | 内容 |
|---|---|---|
| RAID 0 | ストライピング | データを複数ディスクに分散。高速だが冗長性なし（1台故障で全滅） |
| RAID 1 | ミラーリング | 同じデータを2台に書き込む。1台故障まで復元可能。容量は半分 |
| RAID 5 | パリティ分散 | データ＋復元用パリティを分散。1台故障まで復元可能 |
| RAID 6 | デュアルパリティ | RAID 5のパリティを2重化。2台同時故障まで復元可能 |
| RAID 10 | ミラー＋ストライプ | RAID 1＋RAID 0の組み合わせ。高速かつ冗長性あり。必要台数は多い |

**仮想化・ネットワーク効率化**
- **仮想サーバー**：1台のサーバーに論理的に複数サーバーの機能を持たせる。障害対策に有効だがスペックを分け合う（クライアントPC版はVDI）
- **ファブリックネットワーク**：サーバーとスイッチを網目状に構成し、複数サーバー間の通信を効率化

### 運用管理

- **SLA**：システム運用のサービスレベルを定義する指標
  - **可用性**：稼働率の保証値。99%なら年間約88時間、99.9%なら約9時間のダウンタイムが許容される
  - **MTTR**（Mean Time To Repair）：障害発生後、平均何時間以内に復旧できるかの指標
